# チャネル別 仕様（列マッピング・シート名・出力先・既知の差異）

型は共通でも、列の位置・シート名・データ列数はチャネルごとに違う。
ここは「お手本」として使うための記録。**実際に作るときは対象の.xlsmで現物を再確認すること**
（モール側の仕様変更で列がずれることがあるため、ここを盲信しない）。

## Amazon

- マスター: `Amazon_Module_マスター.bas`（VB_Name = "Amazon_マスター"）
- 衣類判定: `Amazon_Module_衣類判定.bas`（関数 `Is衣類Amazon`、キーワード12語）
- 最終行判定列: **C列（order-id）**
- 発送方法自動判定で参照する列:
  - 商品名 = **N列**（product-name）
  - 送付先氏名 = **S列**（recipient-name）
  - 住所1 = **T列**（ship-address-1）
  - 同一送付先カウント = `CountIfs(S列, 氏名, T列, 住所1)`
- 振分: 編集 → ペット/衣類/佐川 の各「_編集から貼付」へ。
  - C列〜AB列にAmazon注文26列をコピー、AG列(is-prime)→AC列。編集の最終列=35(AI列)。
- 実行3 = 手順6_通知用データ作成 → 手順7_通知番号間違いチェック → 手順8_通知用新規ファイル作成
- 出力後の案内: Amazon管理画面「発送完了報告データ」へCSVアップロード

## 楽天fuu

- マスター: `楽天_Module_マスター.bas`（VB_Name = "楽天_マスター"）
- 衣類判定: `楽天_Module_衣類判定.bas`（**現状は関数 `IsClothing`**、キーワード5語のみ）
- **実物の現物（社長が実運用で使うファイル）**: `デスクトップ\史朗\各種フォーマット\楽天fuu出荷データ作成用フォーマット2026.6.16.xlsm`
  （実行1/2/3ボタン付き。手順1〜7ボタンも併存。Module9の手順2bに「列を全表示してからコピー」の不具合修正入り）
  ⚠️ **`2025_3_10.xlsm` は古い別物。触らない**（実行ボタン無し・手順2bの修正も無い）。修正は必ず2026.6.16版に当てる。
- 最終行判定列: **D列（注文番号）**
- 発送方法自動判定で参照する列:
  - 商品名 = **W列**
  - 同一送付先カウント = `COUNTIFS` を数式でB列に展開（送付先姓+名+都道府県+郡市区）
- 特徴的な点:
  - 衣類佐川のしきい値が `cnt >= 7`、衣類クリック複数が `cnt > 3` と、楽天だけ独自の段階を持つ。
  - ペット判定が「ごはん/おやつ を含み、かつ 3kg/5kg/10kg を含まない」→ クリック、という商品名ルール。
  - 振分は手順2b（DistributeByCategoryでオートフィルタ→可視行コピー）で実装。
- 出力:
  - 佐川: `…\Dropbox\ネットショップ\【出荷】fuu楽天\fuu楽天_佐川E飛伝\fuu楽天_佐川E飛伝_YYYY_MMDD.csv`
  - 通知: `…\Dropbox\ネットショップ\【出荷】fuu楽天\fuu楽天_発送通知\fuu_Rakuten_YYYY_MMDD.csv`
  - パスは `Environ("USERNAME")` でユーザー名を埋める（PCが変わっても動く）。

## Wowma（AUPay Market）

- マスター: `Wowma_Module_マスター.bas`（VB_Name = "Wowma_マスター"、`Is衣類Wowma` も同ファイル内）
- 衣類判定: マスター内に同居（キーワード12語）
- 最終行判定列: **D列（orderId）**
- 発送方法自動判定で参照する列:
  - 商品名 = **G列**（itemName）
  - 送付先氏名 = **K列**（senderName）
  - 住所 = **N列**（senderAddress）
  - 同一送付先カウント = `CountIfs(K列, 氏名, N列, 住所)`
- 振分: 貼付先のA列=発送方法・B列=分類、D列〜にWowma22列（C列起点でResize 22）をコピー。編集最終列=30。
- 実行3 = 手順6_追跡番号を通知用データに貼付 → 手順7_通知番号間違いチェック → 手順8_通知用新規ファイル作成
- 出力後の案内: AUPay Market管理画面へアップロード

## ペライチ

- マスター: `ペライチ_Module_マスター.bas`（VB_Name 系、`Is衣類ペライチ`）
- 構造: `実行1_ペライチ_発送方法自動判定` / `実行2_ペライチ_出荷CSV作成` / `実行3_ペライチ_配送情報CSV作成`
- 補助: `シートクリア`, `ペライチ_編集シート転記`, `ペライチ_納品書作成` を内部に持つ
  （ペライチは納品書作成まで含むため、他チャネルより手順が一体化している）
- 列・シート名は対象.xlsmで要確認。

## ふるさと納税（小牧市）

- マスター: `ふるさと納税_Module_マスター.bas`（VB_Name = "ふるさと納税_マスター"）
- 衣類判定: **無し**（謝礼品はすべてペット用品のため）。代わりに `Is佐川対象`（重量・サイズ判定）を持つ。
- 抽出元の現物: `各種フォーマット\★ふるさと納税出荷　フォーマット.xlsx`（※.xlsm化前のテンプレ。マクロは新規）
- シート: `原本`(元データ) / `編集`(作業) / `クリック`(クリックポスト) / `e飛伝3`(佐川) / `物出しリスト`(自動作成) / `納品書`(自動作成)
- **型の差分**：ペット/衣類の区別が無く、発送方法は「クリック」「佐川」の2択のみ。
- 原本の列: 配送No=B / 謝礼品=D / 配送姓=G・名=H / 都道府県=K / 郵便番号=L / 住所1=M / 住所2=N / 電話=O（最終列AN）
- 編集は原本を1列右へずらしてコピーし、A列=発送方法（編集列 = 原本列+1）。
- 振り分けは既存の数式に頼らず**値で直接転記**（クリック宛→クリックシート、佐川宛→e飛伝3）。
- 実行2で **物出しリスト**（発送方法ごとに並べたピッキング表）と **納品書**（寄附者ごと1ページ）も自動作成。
- 送り主は **株式会社SEED で固定**（旧テンプレの「株式会社ARCS／古雅３－６１－４」は前任者の誤り）。

  TEL 0568-47-5090 ／ 事業者番号 T5180001078443（出典: SEED様_納品書フォーマット）。
- 内容品: 「小牧市ふるさと納税謝礼品」。納品書は寄附のお礼のため金額表示なし。
- **未確定（3割）**：実行3（ポータルへの出荷通知CSV）の仕様。`Is佐川対象` の判定キーワード（3kg等の仮置き）。

## チャネル横断の既知の差異（標準化候補）

新規・修正のときに「揃えると品質が上がる」ポイント。会社方針の「3割改善」をここで積む。

| 項目 | Amazon | 楽天 | Wowma | ペライチ | 標準化の方向 |
|---|---|---|---|---|---|
| 衣類判定の関数名 | Is衣類Amazon | **IsClothing** | Is衣類Wowma | Is衣類ペライチ | `Is衣類<チャネル>` に統一 |
| 衣類キーワード数 | 12語 | **5語** | 12語 | 要確認 | 12語の標準リストに統一 |
| 衣類判定ファイル | 別ファイル | 別ファイル | マスター同居 | マスター同居 | どちらでも可。別ファイルの方が再利用しやすい |
| 佐川しきい値 | 手動のみ | 自動(cnt>=7) | 手動のみ | 要確認 | 自動化は便利だが、コスト最適は人の判断を残す |
| 振分の実装 | Select Case で行コピー | オートフィルタで可視行コピー | Select Case | 転記関数 | どちらも可。挙動が同じになることを1件テストで確認 |

> メモ: これらは「直すと良い」候補であって、いきなり全部直す必要はない。
> 触ったチャネルのついでに1つずつ標準へ寄せ、必ず1件テスト→社長確認の順で進める。
